# Authoring a CyanPrint template

A template asks prompts and returns a pure Cyan object describing which processors, plugins, resolvers, and sub-templates to run. Templates stay declarative — no file I/O in `cyan.ts`; the CLI loads archive files and executes every artifact locally.

How the pieces fit: **templates** describe output, **processors** transform scoped template files (e.g. `cyan/default` renders variables), **plugins** post-process the merged output and may run commands, **resolvers** fold same-path conflicts into one file. This project itself was generated by the **meta template** `cyan/new` — the official template that scaffolds new artifacts — and `cyanprint update . --template cyan/new` pulls newer scaffolding (skills, CI, vendored types) later.

## Signature

```ts
export default async function cyan(prompt, ctx) {
  const project = await prompt.text('project', 'Project name', { default: 'my-app' });
  return {
    processors: [
      {
        name: 'cyan/default',
        files: [{ root: 'template', glob: '**/*', type: 'Template' }],
        config: { vars: { PROJECT: project } },
      },
    ],
  };
}
```

Any valid export form loads (const arrow, function expression); the default-function declaration above is the convention.

## Prompts and shared answers

`prompt` supports `text`, `confirm`, `select`, `multiselect`, and `number` — each takes a stable answer key, a message, and an optional default. Write messages as friendly questions ("What is your URL?"), not key names.

Polish and guard-rails, all optional:

- `description` — dim help text rendered below the input (matching select/multiselect option help).
- `placeholder` — a dim ghost value shown inside empty free-form inputs (`text`, `number`).
- `validate` (all except `confirm`) — return `true` or an error message; interactive users get re-prompted inline, headless answers fail the run with your message.
- select/multiselect options may be objects `{ value, label?, description? }` — the description renders below the list and follows the highlighted option.

```ts
const url = await prompt.text('url', 'What is your project URL?', {
  placeholder: 'https://example.com',
  description: 'Used as the homepage link in the generated README.',
  validate: value => (/^https?:\/\//.test(value) ? true : 'Enter a full URL, starting with http(s)://'),
});
const flavor = await prompt.select('flavor', 'Which flavor do you want?', {
  options: [
    { value: 'minimal', description: 'Just the essentials; add your own tooling.' },
    { value: 'batteries', description: 'CI, tests, formatter, and docs preconfigured.' },
  ],
  default: 'minimal',
});
const port = await prompt.number('port', 'Which port should the dev server use?', {
  placeholder: '3000',
  validate: value => (value >= 1024 && value <= 65535 ? true : 'Port must be 1024-65535.'),
});
```

Answers are **shared across the whole composition**: sub-templates generate first, so when a deep dependency asks a question, this template (and its ancestors) reuse that answer instead of re-asking. Use the same answer key for the same concept everywhere. In headless runs (`--headless --answers file.json`) every prompt must be answered by the file or a default.

## Deterministic state

`ctx.deterministic.get(key)` / `ctx.deterministic.set(key, value)` store random-but-stable values (generated ports, IDs, colors). The state is shared across the composition and persisted to `.cyan_state.yaml`, so `cyanprint update` regenerates the same values. Never derive output from timestamps or raw randomness — route it through deterministic state:

```ts
if (ctx.deterministic.get('port') === undefined) {
  ctx.deterministic.set('port', 3000 + Math.floor(Math.random() * 1000));
}
const port = ctx.deterministic.get('port');
```

Parents can also seed a descendant's deterministic state through presets (see below).

## Template files and the default processor

Files live under `template/`. Processors and plugins receive them through **file scopes**: `{ root, glob, exclude, type }`. `type: 'Template'` decodes UTF-8 text for transformation; `type: 'Copy'` copies bytes untouched — use Copy for images, fonts, and other binary assets.

The official **default processor** `cyan/default` renders both file contents and file _paths_ with variable substitution (Eta-based). Config: `vars` (the substitution map) and optional `parser.varSyntax` tag pairs. Unknown placeholders are left untouched, so literal double-underscore tokens in your files survive.

## Composition (sub-templates)

Two ways to include child templates:

- **Static** — list them in `cyan.yaml` under `templates:`. They run **before** this template's `cyan.ts`, and their answers bubble up for reuse.
- **Dynamic** — return `templates: [{ kind: 'template', owner, name, config: { answers: {...} } }]` from `cyan.ts`. They run **after** your script; your answers flow down to them (but theirs don't retro-fill yours).

Rules the engine enforces:

- **Global uniqueness** — each template (`owner:name`, version ignored) may appear only once in the entire tree. Two branches pulling the same template is a hard error; share processors/plugins/resolvers instead.
- **Cycle detection** — self-referential compositions fail fast.
- Every returned artifact must be declared in `cyan.yaml` (`undeclared_artifact` otherwise).

A `kind: template-group` artifact is a template whose only job is composing children.

## Presets — configure descendants (root wins)

`presets.templates` in `cyan.yaml` presets answers and deterministic state for any descendant, transitively:

```yaml
presets:
  templates:
    acme/auth:
      answers:
        provider: github
      deterministic:
        auth-port: 4180
```

Presets cascade through intermediate templates, and when ancestors disagree the **outermost (root) template wins**. Use this to build opinionated wrappers around reusable templates.

## Merge and resolvers

When two layers emit the same path: if both sides agree on exactly **one resolver** (same config) for that path, the resolver merges them; otherwise **last writer wins** and a conflict is recorded. Scope a resolver with `config: { paths: ['README.md'] }` or leave it unscoped to apply everywhere. Binary files are never resolver-merged (LWW + recorded conflict).

Declare resolvers for every path that siblings both write — and for files users will edit (see updating below).

## Post-generation commands

Return `commands: [{ command: 'bun', args: ['install'], allow: true }]` to run setup after files land. Commands without `allow: true` are skipped with a warning. Keep them idempotent. A failing command still writes `.cyan_state.yaml`, so the project remains updatable.

## Author for `cyanprint update`

Users download this template and later run `cyanprint update <project> --template <ref>` — a three-way merge of the prior output, the user's current files, and the new output. Design for it:

- Keep output **deterministic**: same answers + deterministic state ⇒ byte-identical output. Nondeterminism shows up as phantom conflicts on every update.
- Attach a resolver (e.g. `cyanprint/keep-user`) to files users are expected to edit, so updates merge instead of conflicting into `.cyan_conflicts/`.
- Separate template-owned files from user-owned files; document ownership in the README.
- Never remove answer keys between versions — old state must still satisfy new prompts (add defaults for new questions).

## Debugging

- `cyanprint try <dir>` — generate into a scratch folder.
- `cyanprint trace <dir> --headless` — per-file provenance (which template won each path and why: added / resolver-merged / lww-override), each template's isolated output, and diffs of each contribution vs the final result. Point it at a **generated project** instead and it reuses the project's recorded answers and deterministic state (`--template` overrides the recorded ref).
- `cyanprint create` prints each generation step live (template → processors → plugins → resolvers → commands), so slow or failing artifacts are easy to spot.

## Rules

- Deterministic given the same answers + deterministic state.
- Declare every dependency in `cyan.yaml`; keep `cyan.ts` pure data.
- Cover answer combinations in `cyan.test.yaml` (see the testing skill).
