# Updating this artifact

This project was generated by the **meta template** `cyan/new` — the official CyanPrint template that scaffolds artifacts. `cyanprint update` floats it to the latest version and runs a real git three-way merge (base = the old version re-executed from your recorded answers; theirs = the new version; ours = your files on disk), with rename detection:

- Files you never touched update cleanly.
- Files you edited that upstream did not change are kept.
- Files BOTH sides changed either merge line-by-line or keep standard in-file `<<<<<<<` conflict markers; the command exits non-zero listing conflicted files.

## Practice

- Run `cyanprint update .` to pull the latest scaffolding — CI workflows, the CI script, these skills, and the vendored `@cyanprint/sdk` types (`--interactive` to pick versions; `--template cyan/new` to target just this template). Let update own those files; do not hand-edit them, or you will create update conflicts.
- You own: `src/`, `cyan.yaml` metadata, your `package.json` dependencies, and your tests. Bump those yourself (e.g. `bun update`), then re-run `cyanprint test`.
- Your original answers and deterministic state persist in `.cyan_state.yaml`, so update reuses them without re-asking (only new questions prompt).
- After any change, run `cyanprint test` — every case must pass and validations must exit 0.
- If files conflict, resolve the `<<<<<<<` markers in place, then re-run `cyanprint update .` — while markers are pending, `.cyan_state.yaml` stays at the old version, and the re-run completes the move. Finish with `cyanprint test`.
